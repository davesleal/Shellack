Session Lifecycle

From trigger to completion

1. TRIGGER
@SlackClaw run: <task> in #project-channel
→ SlackSession created, thread opened with 🔵 status
→ Backend spawned (Max subprocess or API stream)

2. OUTPUT CHUNKING
Backend produces output continuously
→ Buffer fills until: tool call completes │ paragraph ends │ 3s timeout
→ Chunk posted as new thread message
→ Next chunk edits same message if <5s gap, else new message

3. INPUT — two paths

Path A: Structured (buttons)
Claude posts Block Kit with choices
→ User clicks → answer fed to backend → session continues

Path B: Free-form (typed reply)
User types in thread → bot_unified sees thread reply
→ Checks if active SlackSession exists for thread_ts
→ If yes: feeds reply to backend as user message
→ If no: falls back to normal @SlackClaw behavior

4. SESSION END
Backend signals completion (process exits or stream ends)
→ Final chunk posted
→ ✅ status posted top-level to project channel
→ Session cleaned up from memory

5. TIMEOUTS & ERRORS
Idle >30min → session paused, user notified
Backend error → ❌ posted in thread, session cleaned up
User types "stop" or "cancel" → session terminated immediately

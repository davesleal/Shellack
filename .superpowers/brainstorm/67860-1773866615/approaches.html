Slack↔Claude Session: 3 Approaches

Choose the architecture for bidirectional sessions triggered from Slack

A

Long-lived subprocess (recommended)

Spawn claude -p with stdout piped. For follow-ups, re-spawn with --continue. Output streamed as JSON, chunked to Slack. Supports Max sub natively — no API cost. Each turn is a fresh process but conversation state preserved by Claude Code's own --continue mechanism.

Pros

  • Uses Max subscription
  • Full Claude Code tool use (bash, files, git)
  • No API credits
  • Simple per-turn model

Cons

  • ~1-2s startup per turn
  • --continue is project-scoped (one session per directory)
B

API streaming with tool loop

Use the Anthropic API with messages.stream(). SlackClaw manages the conversation history and tool use loop itself. Supports any model. Output streamed token-by-token, batched before posting to Slack. Works without Claude Max subscription.

Pros

  • No Max subscription needed
  • Full control over conversation history
  • Works for open-source users
  • Model-agnostic

Cons

  • API costs per token
  • Must implement tool use loop ourselves
  • More complex to maintain
C

Dual-mode: Max sub + API fallback

Single SlackSession abstraction with two backends: MaxBackend (subprocess + --continue) and APIBackend (streaming API). Configured per-workspace via .env. Same Slack output logic regardless of backend. Open-source users default to API; Max subscribers get zero cost.

Pros

  • Best of both worlds
  • Open-source friendly
  • Single interface, swappable backend

Cons

  • More code to maintain
  • Two backends to test
